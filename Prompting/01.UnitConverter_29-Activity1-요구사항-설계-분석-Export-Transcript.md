**SRP 분석:**
- Parser(구조 분해) / Validator(유효성) / Registry(단위 저장) / Converter(계산) / Formatter(출력) / App(오케스트레이션) — 각각 변경 이유 하나
- 예: CSV 출력 추가 → `CsvFormatter`만 추가, Converter 불변
- 예: 음수 검증 → `InputValidator`만 수정

**OCP 분석:**
- cubit 추가: Registry.register()만 — if/elif 수정 불필요 ✅
- JSON/CSV: Formatter Strategy — 기존 Formatter 미수정 ✅
- 동적 등록: Registry API 활용 ✅

**한계:**
- 길이→질량 등 물리량 변경 시 새 Converter + App 연결 수정 필요
- Parser/Validator 경계 흐리면 SRP 약화
