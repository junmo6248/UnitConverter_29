## 3. 현재 코드 분석

### 3.1 구조

- **클래스 없음** — `UnitConverter.py` 모듈에 `main()` 단일 함수
- **진입점** — `if __name__ == "__main__": main()`

### 3.2 `main()` 처리 흐름

1. `단위:값` 형식 입력 수신
2. `:` 없으면 형식 오류
3. `unit`, `value_str` 분리 → `float` 변환
4. `meter` / `feet` / `yard` → meter 기준 변환 (`if/elif`)
5. meter·feet·yard 3단위 결과 `print`

### 3.3 변환 상수 (하드코딩)

| 상수 | 값 |
|------|-----|
| feet ↔ meter | `3.28084` |
| yard ↔ meter | `1.09361` |

### 3.4 PRD 대비 갭

| 요구사항 | 현재 상태 |
|----------|-----------|
| OCP | ❌ 단위 추가 시 `if/elif` 수정 |
| SRP | ❌ 입력·검증·변환·출력이 `main()`에 집중 |
| 음수 검증 | ❌ 미구현 |
| 설정 외부화 | ❌ 미구현 |
| 동적 단위 등록 | ❌ 미구현 |
| 출력 포맷 선택 | ❌ 고정 텍스트만 |
| 테스트 코드 | ❌ `tests/` 비어 있음 |

## 4. 제안 아키텍처

### 4.1 계층별 클래스

| 계층 | 클래스 | 역할 |
|------|--------|------|
| Domain | `ConversionRegistry` | 단위·meter 기준 비율 저장·조회 |
| Domain | `LengthConverter` | meter 기준 변환 계산 |
| Domain | `InputValidator` | 형식·숫자·단위·음수 검증 |
| Application | `ConvertLengthUseCase` | 검증 → 변환 → DTO 반환 (UI 비의존) |
| Presentation (CLI) | `CliController` | CLI 실행 흐름 |
| Presentation (CLI) | `InputParser` | `단위:값` 파싱 |
| Presentation (CLI) | `OutputFormatter` | Text/JSON/CSV 포맷 (Strategy) |
| Infrastructure | `ConfigLoader` | JSON/YAML 설정 로드 |

### 4.2 OCP·SRP 충족 요약

| 시나리오 | OCP | SRP |
|----------|-----|-----|
| cubit 단위 추가 | ✅ Registry 등록만 | ✅ |
| JSON/CSV 출력 추가 | ✅ Formatter 구현체 추가 | ✅ |
| 음수 검증 추가 | ✅ | ✅ Validator만 수정 |
| GUI 추가 | ✅ Logic 재사용 | ✅ GuiController만 추가 |

### 4.3 UI/Logic 분리

- **Logic:** `ConversionRegistry`, `LengthConverter`, `InputValidator`, `ConvertLengthUseCase`
- **Presentation:** `CliController`, `InputParser`, `OutputFormatter` (CLI) / `GuiController` (GUI 확장 시)
- Use Case는 `ConvertRequest` → `ConvertResponse` DTO만 반환; `input()`/`print()` 없음
